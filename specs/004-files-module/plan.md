# Implementation Plan: Files Module

**Branch**: `004-files-module` | **Date**: 2026-04-13 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `/specs/004-files-module/spec.md`

## Summary

Build an org-scoped file management system with server-side upload (multipart form), secure download via signed URLs, thumbnail generation with SVG fallback icons, and file discovery via paginated list with search, sort, and multi-select filtering. All file operations (upload, list, download, thumbnail, delete) are scoped to the authenticated user's organisation — org isolation enforced at every layer. File metadata stored in Firestore (`/organizations/{orgId}/files/{fileId}`); binary file data stored in Firebase Cloud Storage at `/organizations/{orgId}/files/{uuid}.{extension}`. Client-side list uses TanStack Query with cursor-based pagination (25 items per page). All file operations cascade-delete when organisation is deleted.

Research decisions: [data-model.md](./data-model.md) | [research.md](./research.md)

### Layer Map

#### File List (GET /api/files)

```
Browser (TanStack Query)
    │  initial render                     after mount / URL change / filter update
    │  SSR (server component)             GET /api/files?search=&sort=&cursor=&kinds=
    ▼                                     ▼
app/(platform)/files/page.tsx             app/api/files/route.ts
    │ calls listFilesQuery()                 │ withAuthenticatedContext
    │                                        │ parseQueryParams (Zod validation)
    ▼                                        ▼
data/files/queries/list-files-query.ts      ← same cached function
    │  'use cache'  cacheTag('files-{orgId}')  cacheLife('minutes', 10)
    ▼
data/files/repositories/file-repository.ts
    │  listFiles({ orgId, search, sort, order, kinds, cursor })
    │  Firestore prefix-query on originalName + in-memory kind filter (OR logic)
    ▼
Firestore: organizations/{orgId}/files
    return: { files[], nextCursor }
```

#### File Download (GET /api/file/:id/download)

```
Browser (TanStack Query)
    │  onClick download button
    ▼
app/api/files/[fileId]/download/route.ts
    │ withAuthenticatedContext
    │ verifyOrgOwnership({ fileId, orgId })
    │ admin.storage().bucket().file(path).getSignedUrl()
    ▼
Firebase Cloud Storage (signed URL, 15-min expiry)
    return: { url, expiresIn: 900, fileName }
```

#### File Thumbnail (GET /api/file/:id/thumbnail)

```
Browser (Image tag)
    │  <img src="/api/files/[fileId]/thumbnail" />
    ▼
app/api/files/[fileId]/thumbnail/route.ts
    │ withAuthenticatedContext
    │ verifyOrgOwnership({ fileId, orgId })
    │ if (isImage) → signed Storage URL
    │ else → SVG data URL (kind-specific icon)
    ▼
Firebase Cloud Storage (image files) OR embedded SVG
    return: Content-Type: image/* OR application/json with dataURL
```

#### File Delete (DELETE /api/file/:id)

```
Browser (form submission)
    │  onClick delete button → confirm modal
    │  call deleteFileAction({ fileId, orgId })
    ▼
src/actions/file-actions.ts
    │ 'use server'
    │ verifyOrgOwnership({ fileId, orgId })
    │ admin.firestore().doc().delete()  ← Firestore document
    │ admin.storage().bucket().file().delete()  ← Storage file
    │ revalidateTag(fileCacheTag(orgId))  ← invalidate list cache
    ▼
Firestore + Cloud Storage (atomic delete)
    return: { success: true }
```

#### SVG Icon Generation (non-images)

```ts
// app/api/files/_lib/generate-svg-icon.ts
export function generateSvgIcon(kind: FileKind): string {
  // Return base64-encoded SVG data URL: `data:image/svg+xml;base64,...`
  // Icons per kind:
  const icons: Record<FileKind, string> = {
    pdf: "<svg>...</svg>", // PDF badge icon
    doc: "<svg>...</svg>", // Document icon
    sheet: "<svg>...</svg>", // Spreadsheet icon
    video: "<svg>...</svg>", // Video icon
    audio: "<svg>...</svg>", // Audio icon
    text: "<svg>...</svg>", // Text icon
    other: "<svg>...</svg>", // Generic file icon
  };
  const svg = icons[kind];
  return `data:image/svg+xml;base64,${Buffer.from(svg).toString("base64")}`;
}
```

## Key Design Decisions

| Decision                                   | Rationale                                                                                                  | Impact                                                                                                                     |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| **Server-proxied upload** (FR-001, FR-016) | Single request flow; easier to enforce 50 MB limit and validate MIME type server-side; simpler for clients | Requires Cloud Functions or Next.js route to proxy multipart to Storage; v2 can optimize with signed URLs                  |
| **No API version prefix**                  | Simpler URLs; versioning via HTTP headers or timestamps; matches org preference                            | Implementation must update endpoints from `/api/v1/` to `/api/`                                                            |
| **SVG data URLs for fallback icons**       | Single request; self-contained; efficient for modern frontends; reduces HTTP roundtrips                    | Icons must be pre-generated as SVG strings; base64 encoding required                                                       |
| **In-memory OR filtering** (FR-012)        | Firestore doesn't support native OR queries; client-side filtering acceptable for v1 (< 1000 files)        | Performance acceptable for org-scoped scale; consider Meilisearch/Algolia for enterprise scale (v2+)                       |
| **Cursor-based pagination** (FR-013)       | Handles real-time list mutations gracefully; no offset drift                                               | Requires opaque cursor generation (base64 docId+sortValue); client must preserve cursor state                              |
| **Org-scoped isolation** (FR-006, FR-009)  | Zero data leakage between organisations; enforced at Firestore path + Security Rules                       | Every CRUD operation must verify `orgId` match; cascading delete required on org deletion                                  |
| **Atomic delete** (FR-014, FR-017)         | Transactional consistency; no dangling Firestore docs or Storage files                                     | Implement as batch delete (Firestore) + Storage file delete in sequence; consider Cloud Functions for guaranteed atomicity |
