# Implementation Plan: Store Module — Data Fetching Strategy

**Branch**: `002-store-module` | **Date**: 2026-04-07 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-store-module/spec.md`

## Summary

Next.js 16 and React 19 reserve Server Actions for mutations (create, update, delete). All list queries use SSR with `'use cache'` functions tagged per-org, GET route handlers for client-side re-fetches, cursor-based Firestore pagination surfaced via URL params, and TanStack Query confined to enrichment-status polling and optimistic mutation updates.

Research decisions: [Decision 8 — Data Fetching Architecture](./research.md#decision-8--data-fetching-architecture-ssr--cache--get-api--tanstack-query)

---

## Technical Context

**Language/Version**: TypeScript 5.x · Next.js 16+ App Router · React 19  
**Primary Dependencies**: TanStack Query v5, Firebase Admin SDK, `next/cache` (`'use cache'`, `cacheTag`, `cacheLife`, `revalidateTag`), Zod, HeroUI v3+  
**Storage**: Firestore — cursor-based pagination via `startAfter(sortValue, docId)`  
**Testing**: Vitest  
**Target Platform**: Node.js 22 (server components, route handlers, Cloud Functions v2)  
**Project Type**: Web application — Next.js App Router  
**Performance Goals**: First page load < 2 s (SC-005); search results surfaced < 1 s after 300 ms debounce  
**Constraints**: Firestore prefix-only search; 25 items per page; `cacheComponents: true` required in `next.config.ts`  
**Scale/Scope**: Org-scoped; multi-tenant isolation mandatory at every layer

- [x] **III. Architecture & Domain Design**: Query functions live in `/src/data/stores/queries/` (read-side, not use-cases); Repositories extended for cursor pagination; data flow: `UI → GET API → cached query fn → Repository → Firestore`. Layer order respected.

### Cursor Pagination

Firestore does not support `OFFSET`. All list queries use `startAfter(sortValue, docId)` for efficient keyset pagination:

```
Cursor format:  base64url( JSON({ id: string, sortValue: string | number }) )

URL example:    /api/stores?sort=createdAt_desc&cursor=eyJpZCI6InN0b3JlMSIs...

Repository:     findByOrgPaginated({ sort, namePrefix, cursor, limit: 25 })
  → if cursor: query.orderBy(field, dir).orderBy('__name__', dir).startAfter(sortValue, id)
  → else:      first page (no startAfter)
Response:       { items: Store[], nextCursor: string | null }
  → nextCursor = null when items.length < limit (last page)
  → nextCursor = encodeCursor(items.at(-1)) otherwise
```

### TanStack Query Usage Rules

| Use case                         | `refetchInterval`                                                                                                   | `initialData`                                    |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| Store list (primary query)       | `false`                                                                                                             | SSR result (first page, default sort, no search) |
| Store detail                     | `false`                                                                                                             | SSR result                                       |
| Document list (primary query)    | `false`                                                                                                             | SSR result                                       |
| Document row — enrichment status | `(q) => q.state.data?.context.status === 'completed' \|\| q.state.data?.context.status === 'failed' ? false : 4000` | —                                                |

Only individual document-status queries use polling. List queries are cache-driven.

## Complexity Tracking

| Deviation                                   | Why Needed                                                                                                                                           | Simpler Alternative Rejected Because                                                                                        |
| ------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| Query functions outside `use-cases/`        | Pure cached reads don't need Zod validation, audit, or rate-limit; extending `BaseUseCase` would add dead branches                                   | A thin `QueryUseCase` base would still require no-op `auditDescriptor` / `rateLimitDescriptor` overrides — false complexity |
| GET route handlers alongside Server Actions | TanStack Query needs a stable URL endpoint; calling Server Actions from `queryFn` bypasses the Next.js cache and violates the Server Action contract | Server Actions do not return HTTP cacheable responses                                                                       |

## Scope: This Refactor vs. Complete Feature

**This plan (data-fetching refactor)** focuses exclusively on modernizing how queries are served via `'use cache'` + GET route handlers + cursor-based pagination. It covers:

- ✅ Query layer infrastructure (cache tags, cursor utilities)
- ✅ Store list SSR + GET routes + TanStack Query integration
- ✅ Document list SSR + GET routes + enrichment-status polling
- ✅ Cache invalidation via `revalidateTag()` in mutations

**Out of Scope** (see tasks-archive.md for complete reference):

- ❌ File upload/download mutations (FR-005 to FR-009) — blocked, resume after refactor
- ❌ File management UI components — blocked, resume after refactor
- ❌ Custom JSON CRUD mutations (FR-010 to FR-014) — blocked, resume after refactor
- ❌ Cloud Functions AI enrichment (FR-024 to FR-028) — independent, ready to start after infrastructure (Phase 1)

**Why Split?** The refactor modernizes query patterns without disrupting working mutation code. Mutations and Cloud Functions will adopt the same cache-tag pattern once this refactor is proven.
